**Class 1: TestNoiseToGradingPipeline**

Tests the flow: clean prompt -> noise injection -> (optional intervention) -> grading

1. `test_type_a_noise_preserves_gradeable_code(self, tmp_path)`:
   - Start with a known HumanEval prompt (e.g., `def add(a, b): return a + b` with test `assert add(1, 2) == 3`)
   - Apply `apply_type_a_noise` at 5% error rate with seed=42
   - Verify the noisy version differs from the clean version
   - Grade the CLEAN code against the test (should pass) to verify the grading pipeline is functional
   - This tests that noise generator and grader are compatible

2. `test_type_b_noise_then_sanitize(self)`:
   - Start with a clean GSM8K prompt
   - Apply `apply_type_b_noise` with l1="mandarin" and seed=42
   - Apply `sanitize_prompt` via mock call_fn that returns a cleaned version
   - Verify: noisy text != clean text, sanitized text is the mock return value, flow completes without error

3. `test_prompt_repeater_doubles_content(self)`:
   - Start with clean prompt text
   - Apply `repeat_prompt` from src.prompt_repeater
   - Verify the output contains the original text repeated (doubled)
   - Verify len(repeated) > len(original)

**Class 2: TestDerivedMetricsPipeline**

Tests the flow: DB with experiment results -> derived metrics computation -> quadrant classification

1. `test_full_derived_pipeline(self, populated_test_db)`:
   - Call `compute_derived_metrics(populated_test_db, cr_threshold=0.8)`
   - Query the derived_metrics table from the DB
   - Verify rows exist with correct columns (prompt_id, noise_type, intervention, model, cr, mean_pass_rate, quadrant)
   - Verify at least one row has quadrant="robust" (from the all-pass pattern)
   - Verify at least one row has quadrant="confidently_wrong" (from the all-fail pattern)

2. `test_cr_then_quadrant_classification(self)`:
   - Compute CR for known pass/fail lists using `compute_cr`
   - Classify using `classify_quadrant` with the computed CR and known mean_pass_rate
   - Verify [1,1,1,1,1] -> CR=1.0, mean=1.0 -> quadrant="robust"
   - Verify [0,0,0,0,0] -> CR=1.0, mean=0.0 -> quadrant="confidently_wrong"
   - Verify [1,0,1,0,1] -> CR=0.4, mean=0.6 -> quadrant="lucky"

**Class 3: TestConfigToDatabasePipeline**

Tests the flow: config -> seed derivation -> DB init -> insert -> query

1. `test_config_seed_to_db_roundtrip(self, tmp_path)`:
   - Create `ExperimentConfig()`
   - Use `derive_seed` (from config) with a prompt_id and noise_type to get a deterministic seed
   - Init a temp DB with `init_database`
   - Insert a run record with the derived seed info
   - Query back and verify the record exists with correct values
   - This validates config, db, and seed derivation work together
