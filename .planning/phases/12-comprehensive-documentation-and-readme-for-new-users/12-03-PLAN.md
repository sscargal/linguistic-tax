**Section 3: Interpreting GLMM Output**
- What GLMM is (Generalized Linear Mixed Model with prompt-level random effects on binary pass/fail)
- What the coefficients mean: positive = increases pass probability, negative = decreases
- How to read p-values (with BH correction note)
- The 3-level fallback chain: BayesMixedGLM full -> reduced -> GEE (explain when each triggers)
- Example output with annotations showing how to read each line

**Section 4: Interpreting Bootstrap Confidence Intervals**
- What bootstrap CIs provide (95% CIs for accuracy and stability metrics)
- How to read: [lower, upper] range, if it crosses a threshold, significance
- BCa method with percentile fallback for degenerate data

**Section 5: Interpreting McNemar's Test Results**
- What it tests: prompt-level fragility (clean passes but noisy fails) and recoverability (noisy fails but intervention passes)
- How to read: statistically significant = intervention has real effect at that prompt level
- BH correction: applied per test-type family

**Section 6: Interpreting Kendall's Tau**
- What it measures: rank-order stability between conditions
- Values: +1 = perfect agreement, 0 = no correlation, -1 = perfect disagreement
- Use case: comparing uniform vs targeted noise degradation patterns

**Section 7: Interpreting Consistency Rate (CR)**
- Definition: pairwise agreement across 5 repetitions
- Value range: 0.0 (never agree) to 1.0 (always identical)
- How it's computed: C(5,2)=10 pairs, count agreements, divide
- Bootstrap CIs for CR

**Section 8: Interpreting Quadrant Classification**
- Four quadrants explained with what each means for the prompt:
  - Robust: high accuracy + high CR (reliable correct answers)
  - Confidently Wrong: low accuracy + high CR (consistently wrong)
  - Lucky: high accuracy + low CR (right sometimes by chance)
  - Broken: low accuracy + low CR (unreliably wrong)
- How to use quadrant migration matrices (before/after intervention)

**Section 9: Interpreting Cost Rollups**
- What's calculated: token savings from compression minus pre-processor overhead
- Net ROI: positive = saves money, negative = costs more than raw
- Per-model and per-intervention breakdown

**Section 10: Interpreting Figures**
For each of the 4 figure types (matching FIG-01 through FIG-04):
1. Accuracy degradation curves: x-axis = noise level, y-axis = accuracy, faceted by model/intervention
2. Quadrant plots: each dot = one prompt-condition, position = accuracy x CR
3. Cost heatmaps: rows/columns = conditions, color = net token savings
4. Kendall's tau bars: bar chart of tau values per comparison

**Section 11: Common SQLite Queries**
Ready-to-run queries for common research questions. Use actual column names from src/db.py schema:
1. "Accuracy by noise level": SELECT noise_type, AVG(CASE WHEN passed=1 THEN 1.0 ELSE 0.0 END) as accuracy FROM experiment_runs GROUP BY noise_type
2. "Accuracy by model and intervention": SELECT model, intervention_type, AVG(CASE WHEN passed=1 THEN 1.0 ELSE 0.0 END) as accuracy FROM experiment_runs GROUP BY model, intervention_type
3. "Cost per model": SELECT model, SUM(cost) as total_cost, COUNT(*) as runs FROM experiment_runs GROUP BY model
4. "Most fragile prompts": prompts that pass clean but fail noisy most often
5. "Most recovered prompts": prompts that fail noisy but pass with intervention
6. "CR distribution": SELECT noise_type, intervention_type, AVG(consistency_rate) FROM derived_metrics GROUP BY noise_type, intervention_type

**Section 2: Project Structure Overview**
Brief recap of src/ modules grouped by layer (link to architecture.md for details).

**Section 3: Running Tests**
- Quick: `pytest tests/ -x -q`
- Full: `pytest tests/ -v`
- Coverage: `pytest tests/ --cov=src --cov-report=term-missing`
- By marker: `pytest tests/ -k pilot` (pilot tests), `pytest tests/ -m "not slow"` (skip slow)
- QA script: `bash scripts/qa_script.sh` (comprehensive pre-release check)

**Section 4: Code Conventions**
- Type hints on all Python functions
- Docstrings on all public functions
- American English throughout
- `src.` import prefix for all internal imports
- Module-level logger: `logger = logging.getLogger(__name__)`
- No print() for logging (use logging module) -- exception: CLI user-facing output uses print()
- No hardcoded API keys (use environment variables)
- Fixed random seeds for reproducibility

**Section 5: Adding a New Model Provider**
Step-by-step guide using the existing pattern:
1. Add model strings to MODELS list in src/config.py
2. Add pricing to PRICE_TABLE in src/config.py
3. Add pre-processor mapping to PREPROC_MODEL_MAP in src/config.py
4. Add rate limit delay to RATE_LIMIT_DELAYS in src/config.py
5. Create `_call_newprovider()` function in src/api_client.py following _call_openai pattern (streaming, TTFT/TTLT tracking, token extraction)
6. Add routing in call_model() function (prefix-based: check model string prefix)
7. Add env var to .env.example
8. Add tests in tests/test_api_client.py following TestCallOpenRouter pattern
9. Update QA script if needed

**Section 6: Adding a New Intervention**
Step-by-step:
1. Add intervention string to INTERVENTIONS list in src/config.py
2. Implement intervention function (pure function, no side effects)
3. Add routing in run_experiment.py intervention dispatch
4. Add tests
5. Update experiment matrix generation (scripts/generate_matrix.py)

**Section 7: Test Patterns**
- Test files mirror src/ modules: src/foo.py -> tests/test_foo.py
- Shared fixtures in tests/conftest.py (sample_config, tmp_db_path, mock factories)
- API response mocking via mock factory fixtures (mock_anthropic_response, mock_google_response, etc.)
- Noise determinism tests: same seed = same output
- Use SimpleNamespace with make_args() for argparse namespace mocking

**Section 8: CI and QA**
- QA script: `bash scripts/qa_script.sh` with 6 sections
- Flags: `--live` (include live API tests), `--section N` (run specific section), `--log` (save output)

**Section 9: Commit Conventions**
- Conventional commits style (based on project history patterns)
- test(): new or modified tests
- feat(): new features
- fix(): bug fixes
- docs(): documentation
- refactor(): code cleanup
