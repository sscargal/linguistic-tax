  <action>
Modify src/setup_wizard.py with these changes:

1. Add constant `MAX_TARGETS_PER_PROVIDER = 4` near other constants.

2. Update `_detect_existing_config()` to support multiple targets per provider:
   - Change `models[provider]` from `{"target": model_id, "preproc": model_id}` to `{"targets": [{"target": model_id, "preproc": model_id}, ...]}` format.
   - Parse all target entries per provider, not just the last one.

3. Update `_handle_existing_config()` to display multiple targets per provider in the existing config summary.

4. Add descriptive text to each wizard step in `run_setup_wizard()`:
   - Step 3 (Provider selection): Add text explaining what providers are and why you might choose multiple: "Each provider gives access to different LLM families. Select all providers whose models you want to test. You only need an API key for selected providers."
   - Step 4 (API keys): Already has "API Key Configuration" header — add: "API keys authenticate your requests. Keys are stored in .env (gitignored) and never sent anywhere except the provider's API."
   - Step 5 (Model roles): `_explain_model_roles` already has good text — no change needed.
   - Step 6 (Model selection): Add intro text: "For each provider, select 1-4 target models to test. More models = broader comparison but higher cost. You can add/remove models before saving."

5. Rewrite `_select_models()` to support multiple targets per provider:
   - For each provider, enter a loop that allows adding up to MAX_TARGETS_PER_PROVIDER targets.
   - After selecting the first target + preproc pair, show a sub-menu:
     ```
     Models for Anthropic (Claude):
       1. claude-sonnet-4-20250514 (preproc: claude-haiku-4-5-20250514)

     Options:
       a  - Add another model (3 remaining)
       r# - Remove model (e.g., r1)
       m# - Modify model (e.g., m1)
       d  - Done with this provider
     ```
   - 'a' adds another target (same flow: pick target, pick preproc). Stop at MAX_TARGETS_PER_PROVIDER.
   - 'r#' removes by number.
   - 'm#' re-prompts for that model's target and preproc.
   - 'd' or Enter moves to next provider.
   - Return format changes: each entry still has provider/target_model/preproc_model keys, but there can be multiple entries per provider now (this is already compatible with _build_config_dict and _validate_models which iterate the list).

6. Update `_show_confirmation()` to handle multiple models per provider gracefully in the summary table. The table already iterates `models` list, so it should just work with more entries.

7. Update `_build_default_target_models()` — no change needed since it's only used for defaults (first target per provider).

Key design choice: The models list return format stays the SAME (list of {provider, target_model, preproc_model} dicts). Multiple targets per provider simply means multiple entries with the same provider key. This keeps _build_config_dict, _validate_models, _build_budget_preview, and _show_confirmation working without changes since they already iterate the flat list.
